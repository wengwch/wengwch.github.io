本周 AI 开发与大模型生态的关键词是：代码 agent 平台化、工具发现标准化、模型 API 治理化。GitHub Copilot 集中更新桌面应用、Agent finder、AGENTS.md 代码评审和 MCP 能力；Anthropic 退役旧 Claude 4 模型；OpenAI API 增强搜索、审核和容器计费。Kimi Code/Kimi CLI/Kimi Agent SDK 继续活跃，通义千问、智谱等路线也在把模型能力向开发者平台和云服务沉淀。

<!-- more -->

## 本周摘要

1. AI coding agent 正从聊天助手变成开发工作台。GitHub Copilot app GA，支持从 issue、PR 或 prompt 启动 agent session；Kimi Code、Kimi CLI、Kimi Agent SDK 也在推进本地项目、命令行和可编程 agent 集成。

2. 工具发现和权限治理成为新焦点。GitHub Agent finder 基于开放 ARD 规范，让 Copilot 按任务发现 MCP server、skills、agents 和 tools，而不是把所有工具塞进上下文。

3. 模型 API 进入运营期。Anthropic 旧模型退役、OpenAI Responses API 工具增强、GitHub Copilot 用量指标细化，都说明企业落地 agent 时，迁移、计费、审计和成本控制与模型能力同样重要。

## 重点动态

### Kimi 代码 agent 生态继续活跃

Moonshot AI 的 GitHub 组织页显示，Kimi Code 面向项目和开发流程，Kimi CLI 面向命令行，Kimi Agent SDK 面向可编程集成。Kimi Code 仓库在 6 月 19 日仍有更新。结合 Kimi-Dev-72B 在 SWE-bench Verified 上 60.4% 的公开成绩，开源代码模型的重点已经从通用聊天扩展到代码定位、修改、自反思和 issue resolution。

对中文开发者来说，关键不是只看英文 benchmark，而是验证这些工具能否稳定处理中文 issue、本地依赖源、私有部署和云环境。

### 模型平台继续走云服务路线

阿里云百炼（Model Studio）把模型上下架、API 调用和应用构建放进统一平台；通义千问/Qwen 继续通过开源模型、云 API 和模型社区扩散；智谱 GLM 路线也在强调 agentic coding、长任务和 API 商业化。

不同产品的落地入口正在分化：一类从 GitHub、VS Code、Claude Code、OpenAI Responses API 等开发工具切入，另一类从云平台、知识库、应用搭建和私有化切入。做技术选型时，应同时看工具链成熟度、中文效果、部署合规、价格、服务稳定性和企业系统集成。

### GitHub Copilot app GA

GitHub 在 6 月 17 日宣布 Copilot app 正式 GA，覆盖 macOS、Windows 和 Linux。它支持在独立 branch/worktree 中并行运行 agent session，并提供集成终端、浏览器、diff 审查和 PR 创建流程。

这标志着代码 agent 从"生成代码"走向"可审查、可验证、可接入团队流程"。

### Agent finder 与 ARD

GitHub Agent finder 允许开发者用自然语言描述任务，由 Copilot 从指定 registry 中检索并排序可用资源。它实现了开放 ARD 规范，并支持企业通过 managed settings 限制 agent 可发现和可使用的资源。

MCP 解决"怎么调用工具"，ARD 更像是在补"怎么发现工具、怎么治理工具目录"。

### AGENTS.md 进入 Copilot code review

GitHub Copilot code review 开始读取仓库根目录的 `AGENTS.md`，并把其中的仓库约定、评审偏好和 agent 行为边界纳入反馈生成。团队可以把架构约束、测试要求、代码风格和禁止事项显式写进仓库。

### GitHub MCP server 支持 issue fields

GitHub MCP server 现在可以读写 issue fields。连接它的 AI 工具可以自动创建带 priority、area、dates 等字段的 issue，也可以按字段过滤已有 issue。这让 agent 更适合做第一轮 triage，但也要求团队先定义好字段 schema 和写入权限。

### 模型与 API 更新

GitHub 将 MAI-Code-1-Flash 扩展到 Copilot CLI、Copilot app、Visual Studio、JetBrains、Eclipse、Xcode 等更多入口；Copilot CLI 也支持企业 BYOK 模型。

Anthropic 在 6 月 15 日退役 `claude-sonnet-4-20250514` 与 `claude-opus-4-20250514`，建议迁移到 Claude Sonnet 4.6 和 Claude Opus 4.8。

OpenAI API 6 月更新包括：Responses API 的 Web search 可返回图片结果；Responses API 与 Chat Completions API 支持 moderation scores；eligible container sessions 改为按分钟计费、5 分钟最低计费。

## 研究与趋势

本周研究侧的共同结论是：agent 的难点正在从"会不会调用工具"转向"能不能在长任务里可靠恢复和治理"。

Google DeepMind 的 AI Control Roadmap 把自主 agent 视作需要监控和限制的 insider-risk 系统。MCP 相关论文显示，软件开发占公开 MCP tools 的主要比例，action tools 占比快速上升，风险正在从"答错"变成"做错"。

《ComplexMCP》指出，顶级模型在动态、有状态、工具互相依赖的环境里成功率仍未超过 60%。Kimi-Dev 强调用 Agentless 训练沉淀代码定位、编辑和自反思能力；SWE-AGI 则显示，读代码和理解规格正成为长任务软件工程 agent 的瓶颈。

## 开发者行动建议

1. 为仓库补齐 `AGENTS.md`，写清技术栈、测试命令、代码风格、禁止改动区域和评审偏好。

2. 用同一套真实任务评估 Copilot、Claude、OpenAI、Qwen、Kimi、GLM 等模型和工具，覆盖英文代码库、中文需求、私有部署和云环境。

3. 不要在业务代码里硬编码模型名，保留模型别名、fallback、灰度和迁移告警。

4. 审计 MCP server 和 agent tools，把工具分成只读、写入、外部副作用和高风险四类。

5. 对复杂任务要求 agent 给出可验证结果：diff、日志、测试输出或可复现实验，而不是只看最终文字说明。

## 本周判断

AI coding agent 正进入平台化治理阶段。工具协议、agent 工作台、模型迁移、计费治理、开源模型、CLI/SDK、中文场景和云上落地正在同时推进。短期内，真正拉开差距的不是单次生成质量，而是仓库规则、工具权限、成本预算和验证闭环。
